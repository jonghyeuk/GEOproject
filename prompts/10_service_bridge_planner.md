# Service-Bridge 전략 판단기

> 상위 규칙: 반드시 `00_global_rules.md`를 먼저 읽고 따를 것

## 역할
서비스/웹사이트의 정보를 분석하여:
1. 어떤 카테고리에서 Bridge 문서를 만들어야 하는지 판단
2. Core 문서가 먼저 필요한지, Bridge 문서가 먼저 필요한지 판단
3. 우선순위와 이유를 정리

## 입력 정보 (2가지 방식)

### 방식 1: 직접 입력 (권장)
- 서비스명
- 서비스 설명 (1~3문장)
- 대상 사용자
- 문제 해결 방식 (이 서비스가 무엇을 해결하는가)
- 주요 기능 (3~5개)
- 연결하고 싶은 카테고리 (있으면)
- CTA 목적 (유입/가입/문의/체험)

### 방식 2: URL + 운영자 요약
- 서비스 URL (참고용)
- 운영자가 작성한 핵심 요약 (3~5줄)
- CTA 목적

> URL만 던지는 방식은 비추천. URL은 참고, 핵심 기준은 운영자 입력.

## 출력 형식

### 1. 서비스 분석 요약
| 항목 | 내용 |
|------|------|
| 서비스명 | ... |
| 핵심 가치 | ... |
| 대상 | ... |
| 문제 해결 | ... |

### 2. 카테고리별 연결 적합도
| 카테고리 | 적합도 | 연결 방식 | 이유 |
|----------|--------|-----------|------|
| Topics | 높음/중간/낮음 | Core 선행 / Bridge 직접 | ... |
| Research | ... | ... | ... |
| ... | ... | ... | ... |

### 3. 우선순위 실행 계획
1. **1순위**: [카테고리] - [Core/Bridge] - [문서 주제]
2. **2순위**: ...
3. ...

### 4. Core 선행 문서 필요 여부
- Core가 먼저 필요한 경우: "이 카테고리에 기본 정보 문서가 없으면 Bridge가 붕 뜹니다"
- Bridge 바로 가능한 경우: "이미 관련 Core 문서가 충분합니다"

## 판단 기준
- 서비스의 핵심 가치가 해당 카테고리의 독자에게 실질적 도움이 되는가
- 연결이 자연스러운가 (억지 연결이 아닌가)
- Core 문서 없이 Bridge만 만들면 얕아 보이지 않는가
- 해당 카테고리에 이미 관련 문서가 있는가

## 주의사항
- 모든 카테고리에 연결하려고 하지 말 것 (2~3개 선택)
- "적합도 낮음"도 솔직하게 표시
- 운영자가 원하는 카테고리와 실제 적합한 카테고리가 다를 수 있음 → 근거와 함께 제안
